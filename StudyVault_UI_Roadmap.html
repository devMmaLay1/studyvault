Read + Review

How each page knows what to show

Every page (except index.html) reads URL query parameters to know what data to load. When you click a Level card on the dashboard, it takes you to level.html?lid=100L. When you click a course, it goes to course.html?cid=MTH101. The JS on each page reads these params and renders the right data from your data.js.

level.html?lid=100L semester.html?sid=100L_sem2 course.html?cid=MTH101 module.html?mid=MTH101_m1 unit.html?uid=MTH101_m1_u1

Pattern to follow

How to Build Every Component

All components follow this same pattern — a function that takes data, returns an HTML string, and is injected into a container div.

// 1. In app.js or the page-specific JS file: function renderProgressBar(reviewed, total) { const pct = total ? Math.round((reviewed / total) * 100) : 0; const color = pct >= 80 ? 'green' : pct >= 40 ? 'amber' : 'grey'; return ` <div class="progress-wrap"> <div class="progress-bar ${color}" style="width:${pct}%"></div> </div> <p class="progress-label">${reviewed} / ${total} units reviewed</p> `; } // 2. In HTML, create a container with an id: <div id="course-progress"></div> // 3. In the page's JS file, inject it: const prog = getCourseProgress(course); document.getElementById('course-progress').innerHTML = renderProgressBar(prog.reviewed, prog.total);
Where everything lives

Data & Storage Strategy

Your app has two types of data. Understanding the difference is critical before you write any code.

📄
data.js — Static Content

Permanent academic content. You edit this file in VS Code when you want to add or update summaries. Never changes at runtime.

  • Level IDs, labels, semester structureFixed forever — 100L, 200L, 300L, 400L
  • All 11 course IDs, codes, titlesCIT104, MTH101, PHY102 etc.
  • All 46 module IDs and titlesMTH101_m1 "Set Theory" etc.
  • All 162 unit IDs and titlesMTH101_m1_u1 "Elementary Sets Theory"
  • Unit summaries (summary: "...")You paste ChatGPT summaries here over time
💾
localStorage — User Data

Your personal interactions with the site. Stored automatically by the browser. Persists across sessions. Never touches data.js.

  • reviewed_MTH101_m1_u1 = "true"One key per unit — did you review it?
  • note_MTH101_m1_u1 = "My note here"One key per unit — your personal notes
  • theme = "dark" or "light"Your preferred color theme
  • recent_units = JSON arrayLast 5 unit IDs you visited (for dashboard)
localStorage key naming

localStorage Key Convention

Use this exact pattern for every key so your JS functions work correctly.

// KEY FORMAT EXAMPLE VALUE PURPOSE "theme" "dark" // light or dark mode "reviewed_{unitId}" "true" // e.g. reviewed_MTH101_m1_u1 "note_{unitId}" "Notes text here" // e.g. note_GST102_m4_u3 "recent_units" '["uid1","uid2"]' // JSON array of last 5 visited
CSS custom properties

CSS Variables for Dark Mode

Put all colors as CSS variables in styles.css. Dark mode is then just one attribute toggle — no extra classes needed anywhere.

/* assets/css/styles.css */ :root { --bg: #f7f4ef; /* page background */ --surface: #ffffff; /* card/component background */ --border: #e0dbd2; /* borders and dividers */ --text: #1a1714; /* main text */ --text-muted: #7a7469; /* secondary/muted text */ --accent: #2d5be3; /* primary blue accent */ --green: #16a34a; /* reviewed / success */ --amber: #d97706; /* in-progress warning */ } [data-theme="dark"] { --bg: #0f0e0c; --surface: #1c1a17; --border: #2e2b26; --text: #f0ece4; --text-muted: #7a7469; --accent: #5b8af5; --green: #22c55e; --amber: #fbbf24; } /* Now use only var(--bg), var(--text) etc. everywhere — dark mode is automatic when JS sets data-theme="dark" on <html> */

Before you start coding

Key Rules & Tips

Things that will save you hours of frustration. Read these once before you write any code.

⚠️ Script Order Matters
Always load scripts in this order in every HTML file: data.js first, then app.js, then the page-specific JS. If you put dashboard.js before data.js, nothing will work because academicData won't exist yet.
💡 Build One Page at a Time
Do NOT try to build all 6 pages at once. Fully complete index.html first. Then level.html. Then continue down the chain. If one page breaks, you'll know exactly which file caused it.
✅ Console-First Debugging
Keep browser DevTools open at all times. After every function you write, test it in the console before moving on. getUnitById("MTH101_m1_u1") should return an object, not undefined.
🚫 Never Use innerHTML for User Input
The Notes panel textarea content should be set with textarea.value = text not innerHTML. Only use innerHTML for content that comes from your own data.js file.
⚠️ Always Check for null
Every page reads a URL param. If someone navigates to course.html with no ?cid= param, getParam('cid') returns null. Always handle this: show a "Course not found" message instead of crashing.
💡 Summaries Can Be HTML
The summary: "" field in data.js renders as innerHTML. So your ChatGPT summary can include <h3>, <ul>, <strong>, <p> tags for richer formatting inside unit.html.
✅ Use Document Title for Navigation
Set document.title dynamically in each page's JS. On unit.html it could be "Elementary Sets Theory — MTH101". This makes browser history and tabs readable.
🚫 Don't Skip the Placeholder
On unit.html, if unit.summary === "", show a helpful placeholder: "No summary yet. Open data.js in VS Code and paste a summary into this unit's summary field." This prevents confusion.
⚠️ Mobile First for unit.html
unit.html is the most-visited page. Make sure it reads well on your phone. Use a max-width of ~680px centered, 16–18px font for summary text, and large tap targets for buttons.
💡 Add Loading States
Each page JS runs after DOM loads. Consider adding a brief CSS animation or fade-in so content doesn't "pop" into view. A simple opacity: 0 → 1 transition on the main container looks professional.
✅ Verified Build Order
Phase 0 → 1 → 2 → 3 → 4 → 5 → 6 → 7. Do not skip. After each phase, verify in browser before moving to the next. This prevents mysterious bugs from accumulating.
🚫 Don't Over-Engineer Early
Build it simple first. Get it working first. Style it second. Search in Phase 6, not Phase 2. Dark mode in Phase 2, not Phase 7. Follow the order in the roadmap — it's sequenced deliberately.
